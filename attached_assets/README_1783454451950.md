# Cookie Consent Kit

A self-contained, copy-ready cookie consent banner with Google Consent Mode v2
and Google Tag Manager wiring. Ported from the Built to Hold site so any other
site can get identical behavior:

- Consent defaults **denied** before anything loads (Consent Mode v2)
- Banner with Accept all / Reject all / per-category preferences
- Choice persisted in localStorage; replayed silently on return visits
- GTM only fires consent-gated tags after the visitor grants consent
- A "Cookie Settings" link reopens the banner any time

---

## Before you start: have your GTM container ID ready

You need the **new site's own GTM container ID** (format `GTM-XXXXXXX`).
Find it in Google Tag Manager: **Admin → Container ID** (also shown top-right
in the workspace). Don't reuse another site's container — create one container
per site so tags and triggers stay independent.

**Where the ID goes:**

- **Vite sites (recommended — true single source):** set `GTM_ID` in
  `head-snippet.html` to `"%VITE_GTM_ID%"` and define `VITE_GTM_ID` as an env
  var/secret. `body-noscript.html` already reads the same variable, so the env
  var is the only place the ID lives — swapping containers never touches code.
- **Everything else:** hardcode the ID on the marked `GTM_ID` line in
  `head-snippet.html`. If you also use the optional `body-noscript.html`
  (JS-disabled visitors only), put the same ID there too — noscript can't read
  a JS variable, so those are the only two spots and they must match.

**Common install mistake:** a mismatched head/noscript ID, or a CSP that
blocks GTM (step 5). Check both before debugging anything else.

**Built-in safety:** the loader only runs when the ID starts with `GTM-`.
A missing or mistyped ID means GTM stays completely dormant — no network call,
no console errors. The banner still shows and records consent; it just has no
GTM to gate. So if GTM/GA Preview shows nothing, check this line first.

## What's in the kit

| File | What it is | Where it goes |
|---|---|---|
| `head-snippet.html` | Consent Mode v2 defaults + guarded GTM loader | Inside `<head>`, as early as possible |
| `body-noscript.html` | Optional GTM noscript fallback | Right after `<body>` |
| `cookie-consent.tsx` | The banner (React component) | Your components folder |
| `consent.config.ts` | Per-site storage key + privacy policy URL | Next to the component |
| `cookie-consent.css` | Standalone banner styles (CSS-variable themed) | Your stylesheet pipeline |
| `footer-cookie-settings-link.md` | "Cookie Settings" reopen link snippets | Your footer |

## Install (in this order)

### 1. Paste the head snippet

Copy both `<script>` blocks from `head-snippet.html` into your site's `<head>`.
**The consent-defaults script must stay above the GTM loader and above any
other analytics/marketing tag.** If a tag loads before the defaults run, it can
fire cookies before consent is denied — the whole point of Consent Mode.

### 2. Set the new GTM ID

Replace `GTM-REPLACE_ME` on the marked `GTM_ID` line (or wire the Vite env var,
see above). Optionally paste `body-noscript.html` after `<body>` with the same ID.

### 3. Configure per-site values

Edit `consent.config.ts`:

- `CONSENT_STORAGE_KEY` — **must be unique per site** (e.g. `"acme_consent"`).
  If two of your sites share a key (possible on shared domains/subdomains with
  the same origin), a choice on one silently applies to the other.
- `PRIVACY_POLICY_URL` — the new site's privacy policy link shown in the banner.

Also review the banner copy in `cookie-consent.tsx` (the intro sentence and the
category descriptions) and the theme variables at the top of
`cookie-consent.css` (`--cc-bg`, `--cc-accent`, `--cc-font`, …) so they match
the new site's brand.

### 4. Mount the banner and add the footer link

- Drop `cookie-consent.tsx`, `consent.config.ts`, and `cookie-consent.css` into
  the project and render `<CookieConsent />` once, near the app root.
  Lazy-loading it is recommended (it's not needed for first paint):

  ```tsx
  import { lazy, Suspense } from "react";
  const CookieConsent = lazy(() => import("./components/cookie-consent"));
  // in your root component's JSX:
  <Suspense fallback={null}><CookieConsent /></Suspense>
  ```

- Add the "Cookie Settings" link to the footer (see
  `footer-cookie-settings-link.md`). It dispatches the `open-cookie-settings`
  event the banner listens for.

### 5. CSP — required if the site sends a Content-Security-Policy header

If the new site has a strict CSP, the browser will **silently block GTM**
(no errors visible to users, tags just never load). Allowlist:

| Directive | Add |
|---|---|
| `script-src` | `https://www.googletagmanager.com` (plus `'unsafe-inline'` for the inline head snippets, or use nonces) |
| `img-src` | `https://www.googletagmanager.com https://*.google-analytics.com` |
| `connect-src` | `https://www.googletagmanager.com https://*.google-analytics.com https://*.analytics.google.com` |
| `frame-src` | `https://www.googletagmanager.com` (needed for the noscript iframe and GTM preview) |

No CSP header on the new site? Skip this step.

### 6. Configure GTM itself

In the new GTM container, set your GA4 (and any ad) tags to respect consent:
GA4 tags honor Consent Mode automatically; for others use
**Tag → Consent Settings → Require additional consent** as needed. The banner
also pushes a `consent_update` dataLayer event (with `analytics_consent` /
`ad_consent` values) you can use as a custom trigger to fire tags immediately
after the visitor accepts, without waiting for the next page view.

## Verify it works (end-to-end checklist)

1. **First visit** (fresh incognito window): banner appears; in DevTools console,
   `window.dataLayer` contains a `['consent','default',{analytics_storage:'denied',…}]`
   entry **before** any GTM events.
2. **GTM loads**: `window.google_tag_manager` exists and has your `GTM-…` key.
   (If not: check the `GTM_ID` line, then CSP.)
3. **Accept all**: dataLayer gains `['consent','update',{analytics_storage:'granted',…}]`
   and `{event:'consent_update', analytics_consent:'granted',…}`; localStorage
   has your storage key with `"analytics":"granted"`.
4. **Reject all** (fresh incognito): the update entry says `denied`; no GA
   cookies (`_ga*`) are set.
5. **Persistence**: after accepting, reload — banner stays hidden and a granted
   `update` is replayed automatically.
6. **Reopen**: footer "Cookie Settings" link reopens the banner with the
   preferences panel expanded.
7. **Real delivery**: with consent accepted, confirm hits arrive in GTM Preview
   mode and GA4 Realtime. (Browser wiring can be perfect and a misconfigured
   container still drops hits — this step needs your GTM/GA access.)

## Not using React or Vite?

- **The head/body snippets and CSS are framework-agnostic** — they work on any
  site (WordPress, plain HTML, Next.js, Vue, …) as-is. Only the Vite
  `%VITE_GTM_ID%` templating trick is Vite-specific; hardcode the ID elsewhere.
- **The banner component is React.** Porting it means reproducing four behaviors:
  1. On load: read stored consent; if present, push the consent `update` +
     `consent_update` event; otherwise show the banner.
  2. Buttons: Accept all / Reject all / Save preferences each push the update
     pair and persist `{analytics, ads, ts}` JSON under your storage key.
  3. Show/hide via the `show` / `prefs-open` classes on `#cookie-consent-banner`
     (the CSS handles the rest).
  4. Listen for the window `open-cookie-settings` event to reopen with
     preferences expanded.
- **Next.js note:** the head snippets must render as raw inline scripts before
  hydration (e.g. `dangerouslySetInnerHTML` in the document head or
  `next/script` with `strategy="beforeInteractive"`), keeping defaults first.

## Per-site rename checklist

- [ ] `GTM_ID` in `head-snippet.html` (and `body-noscript.html` if used) → new container ID
- [ ] `CONSENT_STORAGE_KEY` in `consent.config.ts` → unique key for this site
- [ ] `PRIVACY_POLICY_URL` in `consent.config.ts` → this site's policy page
- [ ] Banner copy in `cookie-consent.tsx` → matches this site's tone/legal text
- [ ] Theme variables in `cookie-consent.css` → this site's colors/font
- [ ] CSP header (if any) → GTM/GA domains allowlisted
